Documentation

v1.0.0

Overview

Medical Entity Classifier is a full-stack application for extracting, classifying, and validating medical entities and relations from clinical text. It supports two classification backends:

  • LLM-based — Uses Large Language Models (via LM Studio, vLLM, or any OpenAI-compatible server) for high-quality, context-aware entity extraction.
  • BERT-based — Uses locally fine-tuned BERT NER models for fast, offline classification.

Every extracted entity is automatically validated against a Knowledge Graph (internal TTL ontology files and/or an external SPARQL endpoint like Apache Jena Fuseki). Non-matching entities are saved as pending for human review using a 5C quality scorecard (Correctness, Completeness, Consistency, Congruence, Coverage).

Getting Started

1. Set up an LLM server

1
Install and start LM Studio (or any OpenAI-compatible server). Load a model such as meta-llama-3.1-8b-instruct. Make sure the server listens on 0.0.0.0:1234 if using Docker.
2
Go to Configuration → LLM Servers and add a new configuration:
  • Name: any descriptive name
  • Base URL: http://127.0.0.1:1234/v1 (or http://host.docker.internal:1234/v1 inside Docker)
  • Model: the model identifier shown in LM Studio
3
Click Test Connection to verify. Set the configuration as default.

2. Your first classification

Navigate to LLM Classification → Entity Classification, paste a clinical text (or use one of the example texts), and click Classify Entities. Results will show each detected entity with its type, confidence, and KG validation status.

Tip: You can also drag-and-drop PDF, DOCX, or TXT files in the Document Upload tab for batch processing.

Classification

LLM Entity Classification

The Entity Classification page sends your text to the configured LLM, which identifies medical entities and classifies them using the supported entity types. Each entity receives:

  • Confidence score (0–1) from the LLM.
  • KG validation status: exact_match similar_match low_match not_found
  • Similar KG entities with similarity scores.

Entities not found (or only partially matched) in the KG are automatically saved to Pending Entities for review.

BERT Entity Classification

The BERT Classification page works identically but uses a locally loaded BERT NER model instead of an LLM. BERT is faster and doesn't require a network connection, but needs a fine-tuned model (see BERT Training).

You must load a model from the Manage Models page before BERT classification is available.

Relation Classification

The Relation Classification page extracts relationships between entities (e.g., "Heparin" treats "Pulmonary Embolism"). Relations are defined as source → relation_type → target triples and are validated against the KG, then saved to Pending Relations.

Classify All

Classify All combines entity and relation extraction in a single pass. It uses the langextract library with Google Gemini or the configured LLM to extract both entities and relations, generate an interactive visualization, and auto-import results.

You can enable:

  • Use random KG samples — adds random examples from your KG as few-shot context to improve extraction quality.
  • Use known entities/relations — includes your ontology's known types as guidance.

Compare with (KG backend)

All classification pages include a "Compare with" selector below the input area. This controls which Knowledge Graph backend is queried for entity validation:

OptionDescriptionSpeed
AllQueries both internal TTL files and external SPARQL endpointSlowest (most thorough)
External KGOnly queries the external SPARQL endpoint (e.g., Jena Fuseki)Slow
Internal KG (TTL Files)Only searches the local rdflib entity cache from TTL filesFast
Tip: Use Internal KG for rapid prototyping. Switch to All for thorough validation against large external KGs.

Knowledge Graph

TTL Files (Internal KG)

Place .ttl (Turtle) ontology files in the KnowledgeGraph/ folder. The app loads them on startup and builds an in-memory entity cache for fast matching. You can manage which files are active from KG Overview.

In Docker, this folder is a bind mount from the host — changes are reflected immediately.

External KG (SPARQL Endpoint)

You can connect to an external SPARQL endpoint (e.g., Apache Jena Fuseki) from Configuration → Triple Store. Add a new backend of type SPARQL with the endpoint URL (e.g., http://localhost:3030/medical/query). Enable it and the system will query it alongside internal TTL files during validation.

SPARQL Console

The SPARQL Query page provides an interactive console to run arbitrary SPARQL queries against your configured endpoints. Results are displayed in a table format.

Entity & Relation Review

Pending Items

Entities and relations that don't have an exact_match in the KG are automatically saved to the pending lists:

  • Pending Entities — shows all unvalidated entities with sortable columns, filters, and batch operations.
  • Pending Relations — same interface for relations (source → type → target triples).

From the pending page you can manually approve, reject, or delete entities before they enter the KG.

5C Scorecard

The Entity Review and Relation Review pages evaluate each item on five quality dimensions:

DimensionWhat it measures
CorrectnessIs the text a valid medical term? Checks character validity, length, format.
CompletenessIs the text a viable, non-trivial medical term? Penalizes very short or symbol-only text.
ConsistencyIs the entity type consistent with the text content? Uses LLM or heuristic classification.
CongruenceDoes it match similar entities in the KG? Measures semantic similarity.
CoverageHow well is this entity type represented in the KG? Checks broader ontology coverage.

Each dimension yields a percentage score. An overall weighted score determines the recommendation: Approve, Review, or Reject.

Bulk Review

Both review pages support bulk operations:

  • Review All Pending — evaluates all unreviewed items sequentially with a progress bar. A Cancel button lets you stop at any point.
  • Re-evaluate All — re-scores all items (including previously reviewed ones).

BERT Training

Export IOB Data

Go to LLM Classification → Export IOB to generate IOB-formatted training data from classified texts. This data can be used to fine-tune a BERT NER model.

Train a Model

Navigate to BERT Classification → Train Model. Upload your IOB data, configure hyperparameters (epochs, learning rate, batch size), and start training. Progress and loss curves are shown in real time.

Manage Models

The Manage Models page lets you load, unload, delete, and set a default BERT model. Models are stored in the BERT_models/ folder (bind-mounted in Docker).

Configuration

LLM Servers

LLM Servers manages connection profiles for OpenAI-compatible LLM servers. Each profile stores:

  • Base URL — the server endpoint (e.g., http://127.0.0.1:1234/v1)
  • Model name — the model identifier
  • API key — authentication key (use EMPTY for local servers)
  • Temperature & max tokens — generation parameters
  • Custom prompts — entity prompt, relation prompt, classify-all prompt

One profile can be marked as default and will be used by all classification pages automatically.

Triple Store

Triple Store manages connections to external RDF stores. Supported types:

  • SPARQL endpoint — connects to a remote store like Apache Jena Fuseki.
  • File-based (TTL) — loads a TTL file directly.

Multiple backends can be enabled simultaneously. They are queried in parallel during validation.

Entity / Relation Configuration

The Entity Configuration and Relation Configuration pages let you tune the 5C scorecard weights and thresholds:

  • Weights for each of the 5 dimensions (must sum to 1.0)
  • Approve / Reject thresholds
  • Minimum length, similarity thresholds, etc.

Connecting to LM Studio from Docker

The docker-compose.yml sets DEFAULT_VLLM_BASE_URL=http://host.docker.internal:1234/v1. For this to work, LM Studio must listen on 0.0.0.0:1234 (not 127.0.0.1). Check Settings → Server in LM Studio.

Note: If the LLM server is unreachable from the container, you can still save a configuration by appending ?verify_connection=false to the POST request, or by running the app outside Docker.
